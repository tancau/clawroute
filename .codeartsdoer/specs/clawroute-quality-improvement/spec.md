# HopLLM 质量改进 - 需求规格文档

## 1. 概述

### 1.1 产品定位
基于 HopLLM 现有代码库的审查发现，对国际化、数据完整性、UI 交互、错误处理、代码质量等方面进行系统性改进，提升应用的健壮性、可维护性与用户体验。

### 1.2 改进范围
- **包含**：国际化功能接入、语言切换 UI、数据冗余消除、JSON 数据运行时验证、主题颜色统一、CodeBlock 复制功能、错误边界、组件测试、根目录清理、RuleEditor 组件拆分、模块级状态修复、LICENSE 补充
- **不包含**：新功能开发、UI 视觉重设计、后端服务引入

### 1.3 优先级定义
| 优先级 | 含义 |
|--------|------|
| P0 | 必须修复，影响核心功能或用户体验 |
| P1 | 应当修复，影响代码质量或可维护性 |
| P2 | 建议修复，改善代码规范或项目完整性 |

## 2. 功能需求

### FR-1 国际化功能接入

#### FR-1.1 文案国际化替换
**当** 应用中存在硬编码的中文文本，**系统应** 将其替换为 next-intl 的翻译函数调用，使所有用户可见文案通过翻译文件管理。

**验收标准**：
- Ubiquitous：所有组件中的硬编码中文文本 **应始终** 通过 `useTranslations` Hook 获取
- 翻译 key **应** 与 messages/zh.json 和 messages/en.json 中的 key 一一对应
- 中文翻译文件（zh.json）中的值 **应** 与当前硬编码文本保持一致
- 英文翻译文件（en.json）中的值 **应** 提供准确的英文翻译
- **如果** 某个翻译 key 在 zh.json 中存在但在 en.json 中不存在，系统 **应** 在构建时报错

#### FR-1.2 next-intl 运行时集成
**当** 应用启动时，**系统应** 正确初始化 next-intl 的 Client Provider，使翻译功能在运行时生效。

**验收标准**：
- State-driven：**当** 应用加载后，所有通过 `useTranslations` 获取的文案 **应** 根据当前 locale 正确渲染
- 默认 locale **应为** `'zh'`
- next-intl 的 `NextIntlClientProvider` **应** 在根布局中正确包裹子组件
- 消息文件 **应** 在服务端加载并传递给 Client Provider

---

### FR-2 语言切换 UI

#### FR-2.1 语言切换入口
**当** 用户需要切换界面语言，**系统应** 在 Header 中提供语言切换 UI 组件。

**验收标准**：
- 语言切换组件 **应** 位于 Header 区域
- 语言切换组件 **应** 提供中文和英文两个选项
- 当前语言 **应** 有视觉高亮标识
- 语言切换组件 **应** 在桌面端和平板端均可见

#### FR-2.2 语言切换行为
**当** 用户点击语言切换选项，**系统应** 将界面语言切换为所选语言，并更新所有文案显示。

**验收标准**：
- State-driven：**当** 用户切换语言后，Store 中的 locale **应** 更新为所选值
- **当** locale 更新后，所有使用 `useTranslations` 的组件 **应** 立即重新渲染为对应语言的文案
- 语言选择 **应** 在页面导航后保持（通过 Store 持久化或 URL 参数）

---

### FR-3 数据冗余消除

#### FR-3.1 场景数据去重
**当** scenes.json 中包含与 scene-model-mapping.json 重复的 candidateModelIds 和 defaultTemplateId 字段，**系统应** 消除该冗余，建立单一数据源。

**验收标准**：
- scenes.json 中的 Scene 数据 **不应** 包含 candidateModelIds 和 defaultTemplateId 字段
- 场景与模型的映射关系 **应** 仅在 scene-model-mapping.json 中维护
- Scene TypeScript 类型 **应** 移除 candidateModelIds 和 defaultTemplateId 属性
- 所有引用 Scene.candidateModelIds 或 Scene.defaultTemplateId 的代码 **应** 改为从 scene-model-mapping.json 查询
- 消除冗余后，应用功能 **应** 与改进前完全一致

---

### FR-4 JSON 数据运行时验证

#### FR-4.1 静态数据导入验证
**当** 应用加载静态 JSON 数据文件（models.json、scenes.json、templates.json、scene-model-mapping.json），**系统应** 在运行时验证数据结构符合预期类型，而非仅依赖 TypeScript `as` 断言。

**验收标准**：
- **如果** JSON 数据格式不符合预期类型，系统 **应** 在开发环境下输出明确的错误信息
- **如果** JSON 数据格式不符合预期类型，系统 **不应** 静默忽略错误继续运行
- 验证逻辑 **应** 覆盖所有静态 JSON 数据文件的导入
- 验证逻辑 **应** 在生产环境下同样生效，防止格式错误的数据导致运行时崩溃
- 验证失败时，系统 **应** 提供包含具体字段路径的错误信息，便于定位问题

---

### FR-5 主题颜色统一

#### FR-5.1 硬编码颜色替换
**当** 组件或样式中存在硬编码的颜色值（如 `#0f1117`、`#f8fafc`、`#94a3b8`、`#2a2d3a` 等），**系统应** 将其替换为 Tailwind CSS 主题变量或语义化的 CSS 自定义属性。

**验收标准**：
- 组件代码中 **不应** 出现硬编码的十六进制颜色值（`#xxxxxx` 格式）
- 颜色 **应** 通过 Tailwind 的语义化 class（如 `bg-background`、`text-foreground`、`border-border`）或 CSS 自定义属性引用
- globals.css 中 **可** 定义颜色自定义属性作为唯一颜色值来源
- 替换后，应用视觉效果 **应** 与改进前一致
- 渐变色（如 `from-[#00c9ff] to-[#92fe9d]`）**应** 提取为 CSS 自定义属性或 Tailwind 主题扩展

---

### FR-6 CodeBlock 复制功能

#### FR-6.1 复制交互实现
**当** 用户点击 CodeBlock 组件的复制区域，**系统应** 将代码内容复制到系统剪贴板。

**验收标准**：
- **当** 用户点击复制区域，系统 **应** 调用剪贴板 API 复制代码内容
- 复制成功后，系统 **应** 显示复制成功的视觉反馈（如文案变化或图标变化）
- **如果** 剪贴板 API 不可用，系统 **应** 显示复制失败的提示
- 复制反馈 **应** 在短暂显示后自动恢复为初始状态

---

### FR-7 错误边界

#### FR-7.1 React Error Boundary 接入
**当** React 组件渲染过程中发生错误，**系统应** 捕获该错误并展示友好的错误提示界面，而非白屏。

**验收标准**：
- Ubiquitous：系统 **应始终** 在关键组件层级接入 Error Boundary
- **当** 组件渲染出错时，系统 **应** 展示包含错误说明的降级 UI，而非白屏
- 降级 UI **应** 提供重新加载的操作入口
- Error Boundary **应** 覆盖至少以下区域：配置页整体、规则编辑器、配置预览
- **如果** 可能，错误信息 **应** 包含足够上下文便于开发者定位问题

---

### FR-8 组件测试

#### FR-8.1 React 组件单元测试
**当** 组件包含交互逻辑或条件渲染，**系统应** 提供对应的 React 组件测试，验证 UI 交互行为正确性。

**验收标准**：
- 以下核心交互组件 **应** 有组件测试覆盖：SceneSelector（场景选择）、ModelComparePanel（排序切换）、RuleEditor（规则增删改、拖拽排序）、ConfigPreview（复制与下载）
- 测试 **应** 使用 React Testing Library，以用户行为视角编写
- 测试 **应** 覆盖组件的核心交互路径，而非实现细节
- 测试 **应** 在 `pnpm test` 中可正常执行并通过

---

### FR-9 根目录空目录清理

#### FR-9.1 残留空目录移除
**当** 项目根目录下存在与 src 目录结构重复的空目录，**系统应** 移除这些残留目录。

**验收标准**：
- 根目录下 **不应** 存在空的 app/、components/、data/、lib/、store/、messages/、__tests__/ 目录
- 移除后，项目构建与运行 **应** 不受影响

---

### FR-10 RuleEditor 组件拆分

#### FR-10.1 子组件提取
**当** RuleEditor 组件代码行数超过合理阈值（>200 行），**系统应** 将其拆分为更小的、职责单一的子组件。

**验收标准**：
- RuleEditor 主组件 **应** 委托渲染逻辑给子组件
- 拆分后的子组件 **应** 包含至少：SortableRuleItem（可排序规则项）、DefaultRuleItem（默认规则项）
- 每个子组件 **应** 有明确的 Props 接口定义
- 拆分后，RuleEditor 主组件代码行数 **应** 不超过 100 行
- 拆分后，应用功能 **应** 与改进前完全一致

---

### FR-11 模块级可变状态修复

#### FR-11.1 router-engine 状态隔离
**当** router-engine.ts 中的 `ruleCounter` 为模块级可变状态，**系统应** 消除该模块级状态，使函数在测试中无状态泄漏风险。

**验收标准**：
- router-engine.ts 中 **不应** 存在模块级可变变量（`let` 声明的模块级变量）
- `generateRuleId` **应** 仍能生成唯一 ID
- 多次测试执行间 **不应** 存在状态泄漏（即测试顺序不影响测试结果）
- 重构后，router-engine 的所有现有测试 **应** 仍通过

---

### FR-12 LICENSE 文件补充

#### FR-12.1 开源许可证
**当** 项目为开源项目但缺少 LICENSE 文件，**系统应** 添加合适的开源许可证文件。

**验收标准**：
- 项目根目录 **应** 包含 LICENSE 文件
- LICENSE 文件 **应** 明确声明许可证类型（如 MIT）
- LICENSE 文件 **应** 包含版权年份与版权持有者信息

---

## 3. 非功能需求

### NFR-1 向后兼容性
- 所有改进 **应** 保持应用现有功能不变
- 数据结构变更 **应** 不影响生成的 YAML 配置格式
- API 接口变更 **应** 不影响组件间的调用关系

### NFR-2 渐进式改进
- 各改进项 **应** 可独立实施，不产生强制依赖链
- 每项改进 **应** 可独立验证，不依赖其他改进项的完成

### NFR-3 测试可维护性
- 新增测试 **应** 遵循现有测试框架（Vitest + React Testing Library）
- 测试 **应** 不依赖实现细节，以用户行为视角编写
- 测试 **应** 在 CI 环境中可稳定执行

### NFR-4 构建稳定性
- 所有改进 **应** 不引入构建错误或警告
- `pnpm build` **应** 成功完成
- `pnpm lint` **应** 无新增错误

---

## 4. 约束条件

### 4.1 技术约束
- CON-T1：国际化方案须使用已安装的 next-intl 库，不引入新的国际化依赖
- CON-T2：颜色替换须基于现有 Tailwind CSS 体系，不引入新的样式方案
- CON-T3：错误边界须使用 React 内置 Error Boundary 机制，不引入第三方库
- CON-T4：数据验证方案须为轻量级实现，不引入重型验证库（如 zod），优先使用 TypeScript 类型守卫

### 4.2 改进约束
- CON-I1：每项改进须保持现有功能行为不变（向后兼容）
- CON-I2：改进范围限于审查发现的问题，不扩展到其他优化
- CON-I3：组件拆分须保持组件对外接口不变
